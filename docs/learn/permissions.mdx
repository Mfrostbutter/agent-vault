title: "Permissions"
description: "Understand how instance roles and vault roles control access across Agent Vault."

Agent Vault separates permissions into two independent axes: **instance roles** and **vault roles**. Instance roles control who can manage the Agent Vault server itself (users, agents, global settings, resets). Vault roles control who can operate within a specific [vault](/learn/vaults), approving [proposals](/learn/proposals), managing members, and configuring [credentials](/learn/credentials). The two axes are fully independent, so having one does not imply the other.

## Instance roles

Instance-level roles govern administrative operations across the entire Agent Vault instance. There are two roles: `owner` and `member`.

| Capability                            | Owner           | Member          |
| ------------------------------------- | --------------- | --------------- |
| Manage users (list, remove, set-role) | Yes             | No              |
| Manage agents (list, revoke, rotate, rename) | Yes      | No              |
| List / delete all vaults              | Yes             | No              |
| Send test emails                      | Yes             | No              |
| Reset instance                        | Yes             | No              |
| Access vault contents                 | Only if invited | Only if invited |

<Warning>
  Owners do **not** get implicit vault access. An instance owner must be
  explicitly invited to each [vault](/learn/vaults), just like any other user.
  Ownership grants administrative control over the Agent Vault instance, not
  visibility into vault [credentials](/learn/credentials) or
  [proposals](/learn/proposals).
</Warning>

## Vault roles

Vault-level roles govern what users and [agents](/agents/overview) can do inside a [vault](/learn/vaults) they belong to. There are three roles: `admin`, `member`, and `consumer`.

| Capability                                     | Admin | Member   | Consumer |
| ---------------------------------------------- | ----- | -------- | -------- |
| Use proxy                                      | Yes   | Yes      | Yes      |
| Discover services                              | Yes   | Yes      | Yes      |
| Raise [proposals](/learn/proposals)            | Yes   | Yes      | Yes      |
| View credential names                          | Yes   | Yes      | Yes      |
| Set / delete credentials directly               | Yes   | Yes      | No       |
| Approve / reject [proposals](/learn/proposals) | Yes   | Yes      | No       |
| Manage vault [policy](/learn/policies)         | Yes   | Yes      | No       |
| Invite agents (consumer only)                  | Yes   | Yes      | No       |
| Invite agents (any role)                       | Yes   | No       | No       |
| Invite users to vault                          | Yes   | No       | No       |
| Manage vault users (remove, set-role)          | Yes   | No       | No       |
| Manage agents (revoke, rotate, rename)         | Yes   | No       | No       |
| Delete vault                                   | Yes   | No       | No       |

The **consumer** role is the most restricted, ideal for agents that only need to proxy requests and raise proposals for human review. The **member** role adds the ability to manage credentials, approve proposals, and set vault policy. The **admin** role has full control, including inviting users and agents with any role.

Vault admins and members can both review agent [proposals](/learn/proposals). When an agent raises a proposal, vault members receive the approval link (via email if SMTP is configured) and can approve or reject from the browser or CLI.

## How the two axes interact

The two permission axes are fully independent. An instance owner who is not invited to a vault cannot see its credentials or approve proposals. A vault admin who is not an instance owner cannot manage users or reset the instance.

**Example**: Alice is an instance owner but not a member of the `payments` vault. She can list all vaults and manage users, but she cannot approve proposals or proxy requests through `payments` until she is invited. Meanwhile, Bob is a vault admin on `payments` but a regular instance member. He can approve proposals and manage agents within `payments`, but he cannot create new users or reset the instance.

## The first user

<Steps>
  <Step title="Start the server">```bash agent-vault server ```</Step>
  <Step title="Register">
    The first user to register becomes the instance **owner** and is
    automatically granted **vault admin** on the default vault. ```bash agent-vault
    register ```
  </Step>
  <Step title="Start working">
    The owner can immediately invite agents, set
    [credentials](/learn/credentials), and configure [policies](/learn/policies)
    on the default vault. No further setup is needed.
  </Step>
</Steps>

<Tip>
  A **default** vault is created automatically at startup. The first user is
  granted admin on this vault as part of registration.
</Tip>

## Change roles

### Instance role

Only instance owners can change instance-level roles.

```bash
agent-vault owner user set-role alice@example.com --role owner
```

### Vault role

Vault admins (and instance owners for any vault) can change vault-level roles.

```bash
agent-vault vault user set-role alice@example.com --role consumer --vault my-vault
```

<Warning>
  The last instance owner cannot be demoted. Agent Vault blocks `set-role` if it would
  leave the instance with zero owners.
</Warning>

<AccordionGroup>
  <Accordion title="Owner-level vault operations">
    Instance owners have a small set of cross-vault operations that do not require vault membership:

    - **List all vaults**: `agent-vault owner vault list` shows every vault in the instance.
    - **Delete any vault**: `agent-vault owner vault remove <name>` removes a vault regardless of membership.
    - **Manage agents**: `agent-vault vault agent list`, `info`, `revoke`, `rotate`, and `rename` are owner-only operations that work across all vaults.

    These operations affect vault existence and agent lifecycle, not vault contents. An owner still cannot view credentials, approve proposals, or proxy requests through a vault unless they are an invited member.

  </Accordion>
</AccordionGroup>
