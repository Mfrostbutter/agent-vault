---
title: "How it works"
description: "Walk through a real scenario to understand how Agent Vault handles requests, proposals, and credential attachment."
---

Let's say your agent needs to call the Stripe API. Here's exactly what happens.

## The scenario

<Steps>
  <Step title="Agent discovers available services">
    The agent calls the discovery endpoint to see what's already configured:
    ```bash
    GET {AGENT_VAULT_ADDR}/discover
    Authorization: Bearer {AGENT_VAULT_SESSION_TOKEN}
    ```
    ```json Response
    {
      "vault": "default",
      "proxy_url": "http://127.0.0.1:14321/proxy",
      "services": [],
      "available_credentials": []
    }
    ```
    Stripe isn't listed — the agent doesn't have access yet.
  </Step>
  <Step title="Agent requests access via a proposal">
    The agent creates a proposal proposing access to Stripe:
    ```json
    POST {AGENT_VAULT_ADDR}/v1/proposals
    {
      "rules": [{
        "action": "set",
        "host": "api.stripe.com",
        "description": "Stripe API",
        "headers": {
          "Authorization": "Bearer {{ STRIPE_KEY }}"
        }
      }],
      "credentials": [{
        "action": "set",
        "key": "STRIPE_KEY",
        "description": "Stripe API key",
        "obtain": "https://dashboard.stripe.com/apikeys"
      }],
      "message": "Need Stripe API access for billing",
      "user_message": "I need access to your Stripe account."
    }
    ```
    Agent Vault responds with an approval URL:
    ```json
    {
      "id": 1,
      "status": "pending",
      "approval_url": "http://localhost:14321/approve/1?token=av_appr_..."
    }
    ```
  </Step>
  <Step title="You approve in browser">
    The agent presents the approval URL in chat. You click it, see what's being requested, paste your Stripe API key, and click **Allow**.

    The proposal status changes from `pending` to `applied`. The policy rule and credential are now active.
  </Step>
  <Step title="Agent retries — Agent Vault attaches credentials">
    The agent makes its request through the proxy:
    ```bash
    GET http://127.0.0.1:14321/proxy/api.stripe.com/v1/charges?limit=10
    Authorization: Bearer {AGENT_VAULT_SESSION_TOKEN}
    ```
    Agent Vault:
    1. Validates the agent's session
    2. Matches `api.stripe.com` against the policy
    3. Resolves `{{ STRIPE_KEY }}` → decrypts the stored credential
    4. Strips the agent's auth header
    5. Attaches `Authorization: Bearer sk_live_...`
    6. Forwards to `https://api.stripe.com/v1/charges?limit=10`

    The agent receives the Stripe response. It never saw the API key.
  </Step>
</Steps>

## Key concepts

### Vaults

Isolated containers that group credentials, policies, and agent access together. A default vault is created automatically on first registration.

<Warning>
Owners don't get automatic vault access — they must be explicitly invited like anyone else.
</Warning>

### Credentials

Encrypted credentials stored per vault using AES-256-GCM with an Argon2id-derived key. Agents only see key names (via `/discover`), never values. Agent Vault resolves `{{ credential_name }}` templates at proxy time.

### Policies

Per-vault rules that define which hosts an agent can reach and what headers Agent Vault attaches. Written as YAML with `{{ credential }}` template placeholders.

### Proposals

Agent-proposed bundles of policy rules and credential slots. Each proposal gets a sequential integer ID per vault (like GitHub PR numbers). A human reviews and approves via browser or CLI.

| State | Meaning |
|-------|---------|
| `pending` | Waiting for human review |
| `applied` | Approved and merged into the policy |
| `rejected` | Denied by a human |
| `expired` | Not reviewed within 7 days |

### Sessions

Time-bound, vault-scoped auth contexts (24-hour TTL). Created by `agent-vault vault run`, invite redemption, or agent session minting.

### Agents

Two types:

| Type | Use case | Token lifetime |
|------|----------|---------------|
| **One-time** | Quick tasks, cloud-hosted agents | Single session (24h) |
| **Persistent** | Long-running, CI/CD, always-on agents | Service token (no expiry) + sessions (24h each) |

### Proxy

The `/proxy/{host}/{path}` endpoint that attaches credentials to outbound requests. All `GET`, `POST`, `PUT`, `DELETE`, and `PATCH` methods are supported. The upstream response is returned unmodified.
